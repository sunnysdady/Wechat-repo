# SOP：方案类静态页面项目（从需求到上线的完整模板）

> 本 SOP 从本仓库的实际交付过程提炼（5 次迭代：初版 → 重设计 → 定位重构 → 视觉精修 → 文档化）。
> 目标读者：任何执行模型/执行者。照着做，应能产出同等质量的交付物。
>
> **标注体系**（每一步开头都有）：
> - 🟢 **照做**：机械步骤，严格按写的执行即可，不要发挥。
> - 🟡 **半判断**：有明确规则，但需要在规则内做小决策；拿不准时选保守项。
> - 🔴 **真判断**：决定项目成败的步骤，没有公式。本节会给"判断的方法"而不是"答案"。能力不足时，宁可把这一步的思考过程写出来交给用户确认，也不要假装做了判断。

**适用范围**：把一份"计划/方案/复盘"做成可分享的静态网页并上线。不适用于需要后端、登录、数据库的项目。

---

## 第 0 节：协作总则（🟢 照做，每个会话开始先读）

1. 交付前必须跑一遍第 7 节自审 checklist，问题先修后报。
2. 过程中遇到的选择自己定，理由记下来，最终汇报时集中说明。事先打断用户的只有两类：不可逆操作（删数据、对外发布到新渠道、花钱）和需求范围变更。
3. 所有没验证过的说法，写出来时必须带"未验证/推测"标注。事实性数字（粉丝数、群数、收入）只能来自用户原话，禁止编造、禁止"差不多就行"。

---

## 第 1 节：需求理解与重新定位（🔴 真判断 —— 全流程最重要的一步）

**用户嘴上说的需求，常常不是值得做的需求。** 本项目的实例：用户最初要的是"公众号复更计划"（怎么恢复发文、涨阅读）。但盘点真实资产后发现：这个号的价值不在内容流量，而在它历史上承接过商务对接、手里有存量粉丝和 500+ 微信群。于是把整个方案从"复更计划"重构为"跨境 AI 商务资源入口"——这次重构（commit `5dbfd00`）是整个项目价值最大的一次变更。

**判断的方法**（不是公式，是检查思路）：

1. 列出用户**实际拥有的资产**（不是想要的，是已有的）。
2. 对用户的原始诉求问一句：达成它之后，这些资产被用上了吗？
3. 如果答案是"没用上"或"只用上一小部分"，原始诉求很可能定错了方向——提出重新定位的方案。
4. 重新定位是范围变更，**属于必须和用户确认的事项**（见第 0 节第 2 条）。给出"原方向 vs 新方向"的对比和你的明确推荐，让用户拍板，不要替用户决定。

**能力不足时的降级做法**：如果你不确定自己的重新定位是否靠谱，就按原始诉求做，但在汇报里附上你观察到的"资产与诉求不匹配"的疑点，让用户判断。这比硬做一个错误的重构安全。

> 标注：上述重构的具体当时对话本 SOP 作者未能直接看到，是从 git 历史和 README 反推的（有把握程度：流程主干确定，当时的措辞细节未验证）。

## 第 2 节：信息盘点（🟡 半判断）

开工前，从用户输入里提取并列成清单：

| 类别 | 本项目的实例 |
|---|---|
| 事实性资产 | 存量粉丝、500+ 微信群、过往商务对接经历 |
| 用户的硬约束 | 几乎不增加时间投入，只做拍板动作 |
| 用户的目标 | 30 天恢复入口身份，90 天跑出对接机会 |
| 禁区 | 不许编造数据；标题和结论需用户确认真实性 |

**规则**：
- 每一项标注来源（用户原话/推断）。推断项在交付物里不能写成事实。
- 清单里缺关键信息时，**不要开问**——先用占位符（如"【待确认：具体粉丝数】"）做完整个交付物，把待确认项集中放进最终汇报。一次汇报收齐所有答案，好过来回打断。
- 🟡 判断点：什么算"关键信息"。原则：影响页面结论或对外承诺的算关键，影响措辞细节的不算。

## 第 3 节：技术形态与文件结构（🟢 照做）

**默认技术选型：零依赖单文件 HTML。** 不用框架、不用构建工具、不用 npm。理由：用户双击就能打开、部署是纯静态、AI 迭代时不会碰到环境问题。除非用户明确要求交互功能，否则不要升级技术栈。

固定文件结构：

```
仓库根/
├── outputs/
│   └── <项目名>.html      ← 主页面，全部 CSS 内联在 <style> 里
├── index.html             ← 根路径跳转页（meta refresh 自动跳转 + 可点击链接兜底）
├── vercel.json            ← 部署配置
├── README.md              ← 第 6 节的模板
├── CLAUDE.md              ← 协作规则 + 项目背景
└── .gitignore             ← 至少含 .vercel / node_modules / work
```

`vercel.json` 固定写法（把 destination 换成实际文件名）：

```json
{
  "cleanUrls": true,
  "trailingSlash": false,
  "rewrites": [
    { "source": "/", "destination": "/outputs/<项目名>.html" }
  ]
}
```

HTML 头部固定要求：

```html
<!doctype html>
<html lang="zh-CN">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>账号名｜一句话定位</title>
```

## 第 4 节：内容写作（🔴 真判断，但给了结构模板）

页面信息结构按这个顺序组织（🟢 结构照做）：

1. **Hero 区**：一句话定位 + 核心判断（为什么这么定位）+ 2~4 个关键事实。
2. **现状/资产盘点**：第 2 节清单的可视化呈现。
3. **方案主体**：内容系统/栏目设计，每个栏目说清"是什么、给谁看、用户要做什么"。
4. **执行节奏**：30 天目标、90 天目标，目标必须可检验（"建立线索表"可检验，"提升影响力"不可检验）。
5. **分工边界**：哪些事用户做（拍板类），哪些事 AI 做（执行类）。

**写作的判断点**（🔴 没有公式）：

- **砍内容**：初稿通常会写多。判断标准：删掉这一段，读者的决策会不会变？不会变就删。
- **结论的诚实度**：方案里的预期效果，宁可写保守。用户要拿这页给别人看，吹过头的承诺最后由用户买单。
- **语言风格**：和用户的行业语言一致（本项目是跨境电商圈的说法），不要用咨询报告腔。

**能力不足时的降级做法**：严格按上面 5 段结构填空，每段不超过初稿的 2/3 长度，所有数字加来源标注。结构对 + 不吹牛，就能达到 80 分；剩下 20 分（节奏感、措辞）交给用户改。

## 第 5 节：视觉系统（🟢 照做 —— 直接抄参数）

风格基调：Apple 式极简——大量留白、低饱和配色、细边框、克制的阴影。**不要自由发挥配色**，直接用这套 CSS 变量起步：

```css
:root {
  --bg: #f3f5f2;        /* 页面底色：极浅灰绿 */
  --paper: #fffdf8;     /* 卡片底色：暖白 */
  --ink: #17212b;       /* 正文 */
  --muted: #667085;     /* 次要文字 */
  --quiet: #98a2b3;     /* 弱化文字 */
  --line: #dce3df;      /* 边框 */
  --teal: #0b6f69;      /* 主强调色，每页强调色不超过 2 种常用 */
  --shadow: 0 18px 48px rgba(31, 45, 61, 0.10);
  --radius: 8px;
}
```

硬性规则：

- 字体栈：`"Microsoft YaHei", "PingFang SC", "Segoe UI", Arial, sans-serif`（中文用户，必须中文字体优先）。
- 正文行高 ≥ 1.6；H1 用 `clamp()` 做响应式（参考 `clamp(34px, 4.2vw, 54px)`）。
- 内容最大宽度 ~1280px 居中；必须有移动端断点（≤768px 单列）。
- 卡片统一：1px 边框 + 8px 圆角 + 极淡阴影。禁止重阴影、禁止大渐变、禁止超过 5 种强调色。
- 🟡 判断点仅一个：哪些信息升级成卡片/表格、哪些留作正文。原则：可并列比较的（栏目列表、时间表）用卡片或表格，论述性的用正文。

## 第 6 节：README 与文档（🟢 照做）

README 固定包含这几节，每节 2~6 行：

1. 项目是什么（一句话 + 当前定位，定位变过就写明从什么变成什么）
2. 在线访问（部署 URL；如果目标用户在国内，注明 vercel.app 可能访问不稳的提示和替代方案）
3. 本地打开方式（明确写"无需安装依赖"）
4. 文件清单（每个文件一句话）
5. 当前方案重点（主页面核心结论的摘要，让人不打开页面也知道方案是什么）
6. 协作原则（用户做什么、AI 做什么）

---

## 第 7 节：交付前自审（🟢 照做 —— 不跑完不许汇报）

逐项过，全部通过才能进第 8 节。发现问题：先修，修不了的如实写进汇报。

**A. 机械检查（命令能跑的就跑命令）**

- [ ] HTML 有 `lang="zh-CN"`、charset、viewport 三件套
- [ ] `<title>` 不是占位文字
- [ ] 页内锚点链接（`href="#xxx"`）都有对应 `id`（grep 核对）
- [ ] 没有引用外部 CSS/JS/字体（`grep -E 'src=|href=' | grep http` 检查，仅允许有意为之的外链）
- [ ] index.html 跳转目标路径和实际文件名一致
- [ ] vercel.json 的 destination 和实际文件路径一致
- [ ] git status 干净，没有该提交而未提交的文件

**B. 内容检查（逐条读着核对）**

- [ ] 每个事实性数字都能对应到用户原话；对应不上的已标"待确认"
- [ ] 没有"未验证却写成事实"的句子
- [ ] 30/90 天目标是可检验的表述
- [ ] 用户侧动作只剩拍板类（没有不小心给用户加活）

**C. 视觉检查（用浏览器或截图工具打开看）**

- [ ] 桌面宽度无横向滚动条、无文字溢出卡片
- [ ] 窄屏（375px）下单列正常、字号可读
- [ ] 强调色不超过 5 种，无视觉噪音

**D. 回归检查（仅迭代时）**

- [ ] 这次改动没有破坏上一版已确认过的内容（diff 逐块过一遍）

---

## 第 8 节：汇报模板（🟢 照做）

最终汇报按这个顺序写，一次说完：

```
1. 结果：做了什么，在哪看（URL / 文件路径）。
2. 我替你做的判断：列出过程中自己拍的板 + 各自理由（对应规则 2）。
3. 自审发现并已修复的问题：列出（对应规则 1；没有就写"自审 X 项全过"）。
4. 待你确认的事项：集中列出占位符和待确认项（对应第 2 节）。
5. 不确定的地方：明确标注把握程度（对应规则 3）。
```

## 第 9 节：git 与部署流程（🟢 照做）

1. 在指定分支上开发（没有就从 main 切新分支，命名 `claude/<描述>`）。
2. 提交粒度：一个语义一次提交（参考本仓库历史：初版、重设计、定位重构、视觉精修、文档化各自独立成提交）。提交信息用一句能看懂的英文或中文，说"做了什么"。
3. `git push -u origin <分支名>`，失败按 2s/4s/8s/16s 退避重试最多 4 次。
4. push 后若无 PR 则创建 **draft PR**，PR 描述里贴第 8 节汇报的第 1、2 条。
5. Vercel 是静态托管自动部署，合并到 main 后无需额外操作；改了 `vercel.json` 时部署后要实际访问一次根路径验证 rewrite 生效。

## 附：哪类步骤交给弱模型最危险（给任务分配者看）

| 步骤 | 风险 | 缓解 |
|---|---|---|
| 第 1 节 重新定位 | 错误重构比不重构更糟 | 用"降级做法"：按原需求做+附疑点 |
| 第 4 节 砍内容/结论诚实度 | 弱模型倾向堆砌和夸大 | 强制"每数字带来源"+长度上限 |
| 第 2 节 关键信息判定 | 漏关键项或过度开问 | 占位符机制兜底，汇报时集中收口 |
| 其余各节 | 低 | 照抄模板即可 |
